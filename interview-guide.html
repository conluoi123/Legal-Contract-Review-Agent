Q01 Tại sao chọn LangGraph thay vì LangChain LCEL hay viết loop tay? Trade-off là gì?
HARD★★★★★
▼
Core Answer
LangGraph được chọn vì 3 tính năng không có trong alternatives: native interrupt(), stateful checkpointing, và cyclical graph support.

LangChain LCEL: pipeline tuyến tính (A→B→C), không có cơ chế dừng giữa chừng để hỏi user. Muốn làm HITL phải tự implement message queue + callback, phức tạp và error-prone.

Custom loop: linh hoạt nhất nhưng tự chịu trách nhiệm state management, retry logic, và persistence. Mỗi lần server crash là mất state. Với contract review — process 20-30 giây — crash recovery là yêu cầu thực tế.

LangGraph: graph state persist tự động vào PostgreSQL sau mỗi node. Khi interrupt(), state được serialize. POST /answer resume đúng từ điểm dừng. Không cần viết thêm infrastructure.
Trade-offs
Tiêu chíLangGraphCustom LoopLCEL
HITL (interrupt/resume)✓ NativeManual impl✗ Không có
State persistence✓ PostgreSQLTự xây✗ Không có
Visibility / debug✓ Graph vizKhó debugPartial
Learning curveMediumLow✓ Low
Overhead~15% overhead✓ 0Minimal
Điểm quan trọng cần nêu

Key insight: Câu hỏi thực sự là "bạn có hiểu khi nào KHÔNG cần LangGraph không?" — với task đơn giản không cần HITL, LCEL hoặc loop thường đủ và ít overhead hơn. LangGraph worth it khi có stateful, multi-step workflow với human intervention.

class AgentState:
    contract_id: str
    clauses: List[Clause] = field(default_factory=list)

⚠ Common Mistake

Nếu dùng class Foo: clauses = [] (class attribute), mọi instance share cùng list. Bug cực kỳ khó debug khi chạy concurrent requests — risk list của contract A lẫn vào contract B.

Q03 Nếu server crash khi agent đang chạy giữa chừng, điều gì xảy ra? Recover như thế nào?
HARD★★★★★
▼
Core Answer
LangGraph Checkpointer lưu state sau mỗi node hoàn thành vào PostgreSQL. Khi server crash và restart, ta gọi graph.invoke(None, config={"thread_id": contract_id}) — LangGraph tự load state từ checkpoint gần nhất và resume từ node tiếp theo.

Điều quan trọng: Nodes phải idempotent — chạy lại không gây side effect. Ví dụ: compare_template_node phải upsert vào Qdrant, không insert duplicate. Nếu detect_risks_node chạy 15/30 clauses thì crash, khi resume cần skip 15 clauses đã xử lý.
Idempotency trong Risk Detection
python
def detect_risks_node(state: AgentState) -> AgentState:
    # Track processed clause IDs trong state
    processed_ids = {r.clause_id for r in state.risks}

    for clause in state.clauses:
        if clause.id in processed_ids:
            continue  # Skip đã xử lý → idempotent
        risk = detect_risk(clause)
        state.risks.append(risk)

    return state
Q04 Thiết kế API polling vs WebSocket để theo dõi trạng thái agent. Khi nào dùng cái nào?
MEDIUM★★★☆☆
▼
Phân tích
Polling (dùng cho MVP): GET /status mỗi 2–3 giây. Đơn giản, stateless, dễ implement. Nhược điểm: server nhận nhiều request dù state không đổi, latency cao hơn ~3s.

WebSocket: Server push event ngay khi state thay đổi. Latency gần 0, ít request hơn. Nhược điểm: cần manage connection state, reconnect logic, không work tốt với load balancer (cần sticky sessions hoặc Redis pub/sub).

Server-Sent Events (SSE): Middle ground — server push, nhưng one-directional, simpler hơn WebSocket. Tốt cho "progress stream". FastAPI hỗ trợ native với StreamingResponse.
Decision

MVP: Polling là đủ — process time 15–30s, poll mỗi 2s = 15 requests max. Cost thấp.
Production: SSE để stream progress events real-time — user thấy "Đang phân tích điều khoản 12/30" thay vì loading spinner.

Q05 Làm thế nào để handle concurrent requests — nhiều user upload cùng lúc mà không ảnh hưởng nhau?
HARD★★★★☆
▼
3 lớp isolation
01
Thread isolation via LangGraph thread_id
Mỗi contract có thread_id = contract_id. LangGraph checkpointer isolate state theo thread — không bao giờ cross-contaminate. Giống primary key trong database.
02
Async background tasks (FastAPI BackgroundTasks)
Upload trả về ngay 202 Accepted, agent chạy async trong background. Không block HTTP thread. Với asyncio, 1 worker có thể handle nhiều concurrent agents vì I/O-bound (LLM API calls await-able).
03
Qdrant payload filtering theo contract_id
Mỗi clause embedding được store với payload.contract_id. Query luôn filter theo contract_id — không bao giờ lấy nhầm embedding của contract khác.
Q06 Nếu LLM API timeout hoặc trả lỗi 500, agent xử lý như thế nào? Không được mất kết quả đã có.
HARD★★★★☆
▼
Strategy: Fail-forward, không Fail-stop
Nguyên tắc: lỗi ở 1 clause không được crash toàn bộ pipeline. Dùng try/catch ở node level, ghi lỗi vào state.errors, tiếp tục với clauses còn lại. Report cuối vẫn có giá trị với 29/30 clauses được phân tích.

Retry strategy: Dùng tenacity với exponential backoff (2s, 4s, 8s). Sau 3 lần fail, skip clause đó và log. Không retry vô hạn — tránh làm chậm toàn bộ pipeline.
Code
python
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10)
)
async def call_llm_safe(prompt: str) -> str:
    return await llm_client.complete(prompt)

async def detect_risks_node(state):
    for clause in state.clauses:
        try:
            risk = await call_llm_safe(build_prompt(clause))
            state.risks.append(risk)
        except Exception as e:
            # Không crash — ghi lỗi, continue
            state.errors.append(f"Risk detection failed: {clause.id}: {e}")
    return state

Q07 Giải thích cơ chế interrupt() trong LangGraph. Khi nào thì dùng interrupt_before vs interrupt_after?
HARD★★★★★
▼
Cơ chế hoạt động
interrupt() là một special exception trong LangGraph. Khi node gọi interrupt(payload), LangGraph:
1. Serialize toàn bộ state vào checkpointer (PostgreSQL)
2. Lưu payload vào pending interrupts
3. Return ngay lập tức — graph "suspended"
4. Khi user gọi graph.invoke(Command(resume=answer), config), graph load state từ checkpoint và tiếp tục từ đúng điểm đó, với interrupt() trả về answer.

interrupt_before vs interrupt_after:
— interrupt_before=["node_x"]: dừng TRƯỚC khi chạy node_x. Dùng khi cần user approve trước khi node thực hiện side effect (gửi email, ghi DB).
— interrupt() trong node body: dừng TRONG node, linh hoạt hơn vì có thể conditional (chỉ interrupt khi thiếu data).
Trong dự án này
python
# Dùng interrupt() trong node body (conditional)
def check_missing_entities_node(state):
    missing = state.metadata.missing_critical_fields
    if not missing:
        return state  # No interrupt — continue pipeline

    # Chỉ interrupt khi thực sự cần
    answer = interrupt({
        "type": "missing_metadata",
        "questions": [{"field": f, "message": QUESTIONS[f]} for f in missing]
    })
    # Resume: answer là dict user đã trả lời
    state.metadata.update(answer)
    return state
Q08 Làm thế nào để thiết kế câu hỏi HITL "thông minh" — không hỏi thừa, không hỏi thiếu?
MEDIUM★★★★☆
▼
3 nguyên tắc thiết kế HITL UX
01
Chỉ hỏi khi thực sự cần
Chỉ interrupt khi thiếu critical field (party_a, party_b, effective_date) hoặc risk item có confidence < 0.7. Đừng hỏi optional fields (governing_law) — gây friction không cần thiết.
02
Gợi ý thông minh dựa trên context
Nếu thiếu effective_date, gợi ý "Hôm nay (04/06/2026)?" thay vì blank input. Nếu thiếu governing_law, gợi ý "Việt Nam" (vì hợp đồng tiếng Việt). Giảm 80% user effort.
03
Batch câu hỏi
Nếu thiếu 3 fields, hỏi cả 3 trong 1 lần interrupt. Đừng interrupt 3 lần riêng biệt — làm phiền user và làm pipeline chậm 3× so với cần thiết.
Q09 Graph state được serialize thế nào để store vào PostgreSQL? Có vấn đề gì với objects không serializable?
HARD★★★☆☆
▼
Cơ chế
LangGraph Checkpointer serialize state thành JSON via msgpack hoặc custom serializer. Vấn đề: List[float] (embeddings 1536 dims × 30 clauses) = ~180KB per checkpoint. Quá nặng.

Giải pháp: Không lưu embeddings trong AgentState. Embeddings được lưu riêng trong Qdrant, query lại khi cần. State chỉ lưu clause_id references.
Design Rule

Rule: AgentState chỉ lưu references (IDs) và metadata nhỏ. Large data (embeddings, raw text, files) sống ở external storage (Qdrant, S3). State = pointers, không phải data.

Q10 Nếu user không trả lời câu hỏi HITL trong 24 giờ, agent xử lý thế nào?
MEDIUM★★★☆☆
▼
Timeout Strategy
Cần scheduled job kiểm tra contracts ở trạng thái AWAITING_INPUT quá 24h. Có 3 options:

Option A — Auto-proceed với giá trị mặc định: effective_date = today, governing_law = "Việt Nam". Ghi chú trong report: "Thông tin này được điền tự động do user không phản hồi."

Option B — Mark EXPIRED, notify user: Gửi email/notification, user phải restart process. Clean nhưng friction cao.

Option C — Partial report: Generate report với những gì đã có, đánh dấu rõ fields bị thiếu. Hữu ích nhất cho user.

Q11 Tại sao không dùng Celery/RabbitMQ thay vì FastAPI BackgroundTasks? Giới hạn của cách hiện tại?
HARD★★★☆☆
▼
Honest trade-off
BackgroundTasks (hiện tại): Task chạy trong cùng process với web server. Nếu server restart (deploy mới), task đang chạy bị kill. Không có retry tự động, không có visibility vào queue depth, không scale ra multiple workers.

Celery + RabbitMQ (production): Task persist trong queue dù server restart. Multiple workers. Retry tự động. Flower dashboard để monitor. Nhưng: thêm 2 infrastructure components (Celery worker + message broker), phức tạp hơn cho MVP.

Justification cho MVP: Contract review process là 15–30s. Khả năng server restart đúng trong window đó rất thấp. LangGraph checkpointer đã handle crash recovery. BackgroundTasks đủ tốt cho single-server MVP.
Key Message

Điểm cộng khi phỏng vấn: Biết giới hạn của lựa chọn mình đưa ra và biết khi nào cần upgrade. "BackgroundTasks cho MVP, Celery khi scale to multiple workers" thể hiện engineering judgment.

Q12 Tại sao Hierarchical Chunking tốt hơn fixed-size chunking cho legal documents?
HARD★★★★★
▼
Vấn đề của fixed-size chunking
⚠ Ví dụ thực tế

Fixed chunk 512 tokens có thể split giữa "Điều 2: Nghĩa vụ bảo mật" và "2.1 Nghĩa vụ của Bên nhận". Khi query "nghĩa vụ bảo mật", chunk trả về không biết đây là Điều 2 hay Điều 5, không thể cite chính xác.

Lợi ích hierarchical
1. Citation chính xác: Mỗi clause có full_path = "Điều 2 > Mục 2.1 > Điểm a". Report có thể cite đúng vị trí.

2. Semantic search đúng unit: Search theo nghĩa vụ → query trả về Clause "2.1 Nghĩa vụ bảo mật", không phải chunk bị cắt ngang câu.

3. Parent context: Khi phân tích risk ở Mục 2.1, ta biết nó thuộc Điều 2 (Nghĩa vụ), không phải Điều 4 (Chấm dứt). Context cha ảnh hưởng đến risk assessment.

4. Missing clause detection: Với hierarchical, ta biết "Điều X về tranh chấp" bị thiếu. Với flat chunks, chỉ biết "đoạn text về tranh chấp" không tìm thấy — không granular.

Q14 Giải thích "Zero Hallucination" constraint trong risk detection. Làm thế nào để enforce nó kỹ thuật?
HARD★★★★★
▼
Vấn đề
LLM rất hay paraphrase — "Bên A không chịu trách nhiệm..." → "Bên A miễn mọi trách nhiệm...". Về nghĩa gần giống, nhưng về pháp lý là khác nhau hoàn toàn. Report pháp lý phải quote nguyên văn.

Kỹ thuật enforce:
1. Prompt: "RULE: Trường quote PHẢI là copy-paste nguyên văn từ input. TUYỆT ĐỐI không paraphrase."
2. Few-shot: Mỗi example có quote nguyên văn — LLM học pattern
3. Post-processing validation: Sau khi nhận RiskItem, check assert risk.quote in original_clause_text
4. Nếu fail → re-call LLM với feedback: "Quote này không tìm thấy trong điều khoản gốc. Hãy copy-paste chính xác."
Code enforcement
python
class RiskItem(BaseModel):
    quote: str

    @validator("quote")
    def validate_quote_verbatim(cls, v, values, **kwargs):
        # original_text được pass vào context khi validate
        original = kwargs.get("context", {}).get("original_text", "")
        if original and v not in original:
            raise ValueError(
                ff"Quote not found verbatim in source. "
                ff"Received: '{v[:50]}...'"
            )
        return v
        # instructor sẽ pass error này cho LLM để tự sửa
Q15 Chain-of-Thought trong few-shot examples khác gì standard few-shot? Khi nào nên dùng?
MEDIUM★★★☆☆
▼
Phân biệt
Standard few-shot: Input → Output trực tiếp. LLM học pattern matching.
CoT few-shot: Input → Thinking (lý luận từng bước) → Output. LLM học cách suy luận, không chỉ output.

Khi nào CoT cần thiết: Khi task đòi hỏi multi-step reasoning — ví dụ: "Điều khoản này có bất lợi cho bên nào? Tại sao?" LLM phải xác định chủ thể, đọc nghĩa pháp lý, xác định risk type, rồi mới kết luận severity.

Trong risk detection: Nếu chỉ dùng standard few-shot, LLM match surface patterns ("không chịu trách nhiệm" → HIGH). Với CoT, LLM học phân biệt "không chịu trách nhiệm về bất khả kháng" (LOW) vs "không chịu trách nhiệm về mọi thiệt hại" (HIGH).

Q17 Giải thích cách detect "Missing Clause" — đây là bài toán hai chiều, không phải một chiều.
HARD★★★★★
▼
Hai chiều query
→
Contract → Template (detect MODIFIED)
Mỗi clause trong contract, tìm match gần nhất trong template. Nếu similarity ∈ [0.6, 0.75) → MODIFIED. Phát hiện điều khoản bị thay đổi so với chuẩn.
←
Template → Contract (detect MISSING) ⭐
Mỗi clause REQUIRED trong template, tìm match trong contract. Nếu không tìm thấy (max similarity < 0.75) → MISSING. Đây mới là chiều quan trọng để phát hiện "hợp đồng thiếu điều khoản phạt".
Tại sao một chiều không đủ
⚠ Nếu chỉ check một chiều (Contract → Template)

Chỉ phát hiện được clauses trong contract có "lạ" không. Không biết template có clauses nào mà contract thiếu. Câu trả lời "F1 ≥ 85% cho missing clause detection" phụ thuộc vào implement đúng cả 2 chiều.

Q18 Cosine similarity threshold 0.75 được chọn như thế nào? Tại sao không phải 0.8 hay 0.7?
MEDIUM★★★☆☆
▼
Empirical process
0.75 không phải con số magic — phải empirically tune trên labeled dataset. Process:

1. Tạo test set: 50 cặp (clause, expected_status) đã annotate thủ công
2. Sweep threshold từ 0.5 đến 0.95 bằng 0.05 increments
3. Tính F1 score cho mỗi threshold
4. Chọn threshold maximize F1 trên validation set

Kết quả điển hình: threshold 0.7 → recall cao nhưng precision thấp (PRESENT false positives). 0.8 → precision cao nhưng recall thấp (bỏ sót PRESENT). 0.75 thường là sweet spot cho bilingual legal text.
Key Point

Trong CV: Nêu "tuned similarity threshold via F1 optimization trên 50-sample dataset" — thể hiện engineering rigor, không phải hardcode random number.

Q20 Qdrant HNSW index hoạt động thế nào? Tại sao approximate nearest neighbor tốt hơn exact search?
HARD★★☆☆☆
▼
HNSW Conceptually
HNSW (Hierarchical Navigable Small World) là graph-based index. Mỗi vector được kết nối với K nearest neighbors. Search bắt đầu từ entry point (top layer), greedy descent xuống các layer, mỗi bước move đến neighbor gần hơn với query.

Tại sao ANN tốt hơn exact: Exact search (brute force) = O(n × d) với n = số vectors, d = dimensions. 10K vectors × 1536 dims = 15M operations per query. HNSW = O(log n) average, trade off ~1% recall để đổi lấy 100× speedup.

Trong use case này: Template collection chỉ có ~100 clauses — exact search cũng OK. Nhưng khi contract_clauses tích lũy 10K+ clauses qua nhiều users, HNSW trở nên critical.

Flaky tests — non-deterministic output

Rule: Unit tests = mock LLM, test business logic. Integration tests = real LLM, test prompt quality. Chạy integration tests trước merge vào main, không phải mỗi commit.

Q24 DeepEval đo Hallucination Rate như thế nào? Metric này có limitations gì?
HARD★★★★☆
▼
Cơ chế DeepEval
DeepEval HallucinationMetric dùng một LLM judge (thường là GPT-4) để đánh giá: "Câu nào trong actual_output không được support bởi context (source documents)?". Score = % câu có factual support.

Trong project này: actual_output = report text. context = contract original text. Metric kiểm tra mỗi claim trong report có nguồn gốc từ contract không.
Limitations quan trọng
⚠ Limitations của metric

1. LLM judge cũng có thể hallucinate. GPT-4 judge đánh giá GPT-4 output — không hoàn toàn objective.

2. Không detect paraphrasing hallucination. "không chịu trách nhiệm" → "miễn trách nhiệm" — LLM judge có thể coi là supported vì nghĩa gần giống. Đây là lý do cần thêm substring check thủ công.

3. Threshold 0.05 là convention, không phải con số được tính toán cho domain pháp lý cụ thể. Với hợp đồng, nên yêu cầu 0%.

Q26 Acceptance criteria "E2E latency < 30s" — bạn measure P50 hay P95? Tại sao quan trọng?
MEDIUM★★★☆☆
▼
P50 vs P95
P50 (median): 50% requests hoàn thành trong thời gian này. Misleading — ẩn tail latency.
P95: 95% requests hoàn thành. Nếu P95 = 45s, 1/20 user phải chờ > 45s — UX rất tệ.

Trong benchmark script: Luôn report P50, P95, P99 và stddev. Target "< 30s" phải là P95, không phải mean. LLM API latency có high variance — một call có thể 5s, call khác 25s tùy server load.
Key insight

Senior signal: Biết phân biệt P50/P95/P99 và biết tail latency quan trọng hơn mean thể hiện bạn đã từng deal với production systems, không chỉ development.

System Design & Scalability 4 câu
Q28 Hệ thống scale như thế nào khi có 1000 concurrent users upload hợp đồng cùng lúc?
HARD★★★★☆
▼
Bottlenecks và giải pháp
01
LLM API rate limits → Queue + backpressure
Anthropic API có rate limits (requests/min, tokens/min). Cần Celery queue với rate limiting — không fire tất cả 1000 LLM calls cùng lúc. Priority queue: trả tiền nhiều hơn → queue priority cao.
02
PostgreSQL connection pool → pgBouncer
1000 concurrent agents × 5 DB queries = 5000 concurrent connections. Postgres default max = 100. pgBouncer pooler ở giữa, reuse connections efficiently.
03
Qdrant → Distributed cluster
Single Qdrant node có throughput limit. Khi scale, dùng Qdrant distributed mode với sharding theo contract_id. Qdrant Cloud handle này tự động.
Q29 Làm thế nào để ensure data isolation giữa các clients (multi-tenancy)? Hợp đồng của công ty A không được lộ cho công ty B.
HARD★★★★★
▼
3 lớp bảo vệ
Layer 1 — API (JWT + ownership check):
Mỗi request phải có JWT với user_id. Mọi endpoint check: SELECT id FROM contracts WHERE id=$1 AND user_id=$2. Không tìm thấy → 404 (không phải 403, để tránh information leakage).

Layer 2 — Database (Row-Level Security):
PostgreSQL RLS policy: CREATE POLICY user_isolation ON contracts USING (user_id = current_user_id()). Dù SQL bug không check ownership, RLS tự filter.

Layer 3 — Qdrant (payload filter):
Mọi query Qdrant bắt buộc filter {"contract_id": request.contract_id}. contract_id được validate thuộc user. Embedding của contract khác không bao giờ return.
